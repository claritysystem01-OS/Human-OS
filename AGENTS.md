# Human OS Repository Instructions

## System identity

Human OS is an evidence-based inference and judgment system.

It is not primarily:

- a questionnaire
- an interview engine
- a personality test
- a prompt collection
- a report generator

Interviews are one possible evidence source among transcripts, emails, chat logs, CRM notes, written reflections, behavioral records, and other communication evidence.

## Central artifact

The Working Model is the canonical output of inference.

Reports are audience-specific projections of the Working Model. Reports must not perform independent classification or introduce unsupported conclusions.

## Required pipeline

Evidence
→ Evidence Preparation
→ Observations
→ Competing Hypotheses
→ Adversarial Review
→ Working Model
→ Report Projection

## Evidence discipline

Every conclusion must be traceable to:

1. explicit evidence
2. explicit canonical doctrine
3. an explicitly approved experiential correction

The system must separate the subject's own communication from contextual speech written by other people or AI systems.

Primary-subject evidence must be identified and weighted separately.

## Epistemic discipline

The system must distinguish:

- observation
- interpretation
- hypothesis
- contradiction
- confidence
- uncertainty
- prediction
- recommendation

Do not convert limited evidence into false certainty.

Do not hide contradictory evidence.

Do not use report prose as a substitute for a structured Working Model.

## Change control

Do not alter canonical doctrine merely to make implementation easier.

Do not promote a proposed correction into doctrine without explicit developer approval.

Before broad architectural changes:

1. inspect the repository
2. explain the current behavior
3. identify affected files
4. propose a plan
5. wait for approval unless implementation was explicitly authorized

Prefer small, reviewable changes.

Do not modify any existing files.
Do not create any other files.
